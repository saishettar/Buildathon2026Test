1. EXECUTIVE SUMMARY
────────────────────────────────────────────────────────────────────────────────

Tenor is a production-grade, real-time observability platform purpose-built for
AI agent systems. It provides a rich visual interface for monitoring, inspecting,
and optimizing AI agent execution traces — the step-by-step reasoning and action
sequences that LLM-powered agents perform to complete tasks.

The platform renders live execution traces as interactive Directed Acyclic Graphs
(DAGs) that grow in real-time as agent steps execute. Every step — whether an LLM
inference call, a tool invocation, a planning phase, or a final output — is
captured with full metadata including prompts, completions, token counts, costs,
latency, and error details.

Tenor also integrates Claude (Anthropic's LLM) as an embedded AI advisor that
can analyze workflow performance and suggest concrete optimizations like
parallelization opportunities, cost reductions, and error handling improvements.


────────────────────────────────────────────────────────────────────────────────
2. PROBLEM STATEMENT
────────────────────────────────────────────────────────────────────────────────

As AI agent systems grow in complexity — often involving multi-step chains of
LLM calls, tool invocations, planning loops, retries, and error recovery — teams
lack visibility into:

  • What steps did the agent take, and in what order?
  • How long did each step take? What was the critical path?
  • How many tokens were consumed and at what cost?
  • Where did failures occur and how were they handled?
  • Are there opportunities to parallelize, cache, or optimize?

Existing logging and monitoring solutions are not designed for the DAG-like,
branching nature of agent workflows. Tenor solves this by providing purpose-built
observability with real-time streaming, interactive graph visualization, and
AI-powered analysis.

5. DATA MODEL (UNIVERSAL CONTRACT)
────────────────────────────────────────────────────────────────────────────────

The system uses a "universal contract" — a standardized data schema that works
across any AI agent framework (OpenClaw, LangChain, custom agents, etc.).

5.1 Run

  A "Run" represents a single end-to-end agent execution. It is the top-level
  container for all steps.

  Fields:
    run_id          string (UUID)           Unique identifier
    created_at      string (ISO 8601)       Creation timestamp
    updated_at      string (ISO 8601)       Last update timestamp
    status          enum                    "running" | "completed" | "failed"
    system_type     enum                    "mock" | "openclaw" | "claude" | "other"
    root_step_id    string | null           ID of the first step in the DAG
    metadata        object                  { user_id: string, tags: string[] }

5.2 Step

  A "Step" represents a single unit of work within a run. Steps form a tree
  (parent-child relationships) that is rendered as a DAG.

  Fields:
    step_id               string (UUID)       Unique identifier
    run_id                string (UUID)       Parent run reference
    parent_step_id        string | null       Parent step (null = root)
    name                  string              Human-readable step name
    type                  enum                "llm" | "tool" | "plan" | "final" | "error"
    status                enum                "running" | "completed" | "failed" | "retrying"
    started_at            string (ISO 8601)   When the step started
    ended_at              string | null       When the step finished
    duration_ms           integer             Processing time in milliseconds
    tokens_prompt         integer             Prompt/input token count
    tokens_completion     integer             Completion/output token count
    cost_usd              float               Estimated cost in USD
    input                 object              Raw input data (prompts, tool args, etc.)
    output                object              Raw output data (completions, tool results)
    error                 object | null       { message, stack, code } if failed

5.3 Step Types

  llm     — An LLM inference call (prompt → completion). Colored BLUE.
  tool    — A tool/function invocation (API call, file read, web search). Colored PURPLE.
  plan    — A planning/orchestration step. Colored AMBER.
  final   — The final output of the workflow. Colored GREEN.
  error   — A failed step. Colored RED.

5.4 Step Statuses

  running   — Step is currently executing (shown with pulse animation).
  completed — Step finished successfully.
  failed    — Step encountered an error.
  retrying  — Step is being retried after a failure.

6. BACKEND (FastAPI) — DETAILED BREAKDOWN
────────────────────────────────────────────────────────────────────────────────

6.1 main.py — Application Entrypoint

  The FastAPI app is configured with:
    - CORS middleware allowing connections from localhost:3000 and all origins
    - Lifespan context manager for startup/shutdown logging
    - REST endpoints for CRUD on runs and steps
    - WebSocket endpoint for real-time per-run subscriptions
    - Claude-powered analysis and chat endpoints

  REST Endpoints:
    GET  /api/health                 → { status: "ok", service: "tenor-api" }
    GET  /api/scenarios              → List of available demo scenarios
    POST /api/runs                   → Create a new run (optionally with a scenario)
    GET  /api/runs                   → List recent runs (limit param, default 50)
    GET  /api/runs/{run_id}          → Get a single run by ID
    GET  /api/runs/{run_id}/steps    → Get all steps for a run
    POST /api/steps                  → Create a step manually
    GET  /api/runs/{run_id}/analyze  → Claude-powered performance analysis
    GET  /api/steps/{step_id}/summarize → Claude-powered step summary
    POST /api/chat                   → Send message to Claude optimization advisor

  WebSocket Endpoint:
    ws://localhost:8000/ws/runs/{run_id}
      - Clients subscribe to a specific run's updates
      - Server broadcasts step_update and run_update events in real-time
      - Supports ping/pong keep-alive
      - Auto-cleanup of stale connections

6.2 models.py — Pydantic Data Models

  Defines all enums (RunStatus, SystemType, StepType, StepStatus) and core
  models (Run, Step, RunMetadata, StepError) using Pydantic BaseModel.
  All models use Field(default_factory=...) for UUIDs and timestamps.
  Request models: CreateRunRequest, CreateStepRequest.

6.3 database.py — In-Memory Data Store

  A simple dict-based store keyed by run_id and step_id. Provides:
    - create_run(), get_run(), list_runs(), update_run()
    - create_step(), get_step(), get_steps_for_run(), update_step()
  Designed to be swappable with PostgreSQL via async ORM (SQLAlchemy + asyncpg).
  Instantiated as a module-level singleton.

6.4 websocket_manager.py — WebSocket Room Manager

  Manages WebSocket connections grouped by run_id ("rooms"):
    - connect(run_id, ws) — Accept and add to room
    - disconnect(run_id, ws) — Remove from room, cleanup empty rooms
    - broadcast(run_id, message) — JSON-serialize and send to all room members
    - Auto-removes stale connections on send failure

6.5 simulator.py — Scenario Simulation Engine

  Walks a ScenarioStep tree and emits steps over time to simulate a real agent:
    1. Creates a step in "running" status → broadcasts via WebSocket
    2. Sleeps for the configured delay (simulates processing time)
    3. Updates the step to "completed" or "failed" → broadcasts again
    4. Recursively processes child steps
  Computes costs using configurable per-1K-token rates (GPT-4, Claude 3.5, default).

6.6 scenarios.py — Demo Scenario Definitions

  5 predefined scenario trees, each a nested ScenarioStep dataclass:

  1. flight_booking     — 9 steps: Plan → Search → Analyze → Book → Payment (FAILS)
                          → Retry Payment (succeeds) → Confirm → Final Output
                          Demonstrates: error handling, retries, tool calls, LLM reasoning

  2. research_summarizer — 9 steps: Plan → 3 parallel web searches → Extract findings
                           → Synthesize → Write summary → Final Output
                           Demonstrates: branching, multi-source aggregation

  3. code_assistant      — 7 steps: Plan → Read source → Read test → Analyze error
                           → Search docs → Generate fix → Final Output
                           Demonstrates: code reasoning, file I/O tools

  4. customer_support    — 7 steps: Plan → Lookup customer → Check history → Sentiment
                           analysis → Generate response → Log interaction → Final Output
                           Demonstrates: multi-tool orchestration, sentiment analysis

  5. simple_happy        — 3 steps: Plan → Single LLM call → Final Output
                           Demonstrates: minimal happy path, baseline performance

  Each scenario includes realistic:
    - Step names and types
    - Simulated delays (0.4–3.2 seconds)
    - Token counts (50–2,400 prompt, 30–1,200 completion)
    - Input/output data (prompts, API args, results)
    - Error data (payment declined with stack trace)

6.7 chat.py — Claude-Powered Optimization Advisor

  An embedded AI chatbot that:
    - Receives user messages + optional workflow context (run + steps data)
    - Uses a detailed system prompt instructing Claude to analyze:
      parallelization opportunities, cost reduction, latency improvements,
      error handling, token efficiency, and architecture patterns
    - Formats workflow data (step count, total tokens, costs, duration, failures)
      into a structured context block appended to the system prompt
    - Calls Claude claude-sonnet-4-20250514 via the Anthropic SDK (async)
    - Returns the optimization advice with token usage stats
    - Gracefully handles missing API key with setup instructions

6.8 analysis.py — Claude-Powered Performance Analysis

  Two analysis functions:

  analyze_run(run_data, steps_data):
    - Sends a structured prompt with run statistics and step table to Claude
    - Claude returns JSON with: summary, token_analysis, cost_analysis,
      latency_analysis, error_analysis, recommendations[], and score (0–100)
    - Parses the JSON response, falls back to raw text on parse failure

  summarize_step(step_data):
    - Sends a single step's input/output data to Claude
    - Claude returns JSON with: plain_summary, input_summary, output_summary,
      performance_note (all in plain, non-technical language)

8. REAL-TIME DATA FLOW
────────────────────────────────────────────────────────────────────────────────

The real-time pipeline works as follows:

  1. User clicks "Start Demo Run" → selects scenario from dropdown
  2. Frontend POSTs to /api/runs with { scenario: "flight_booking" }
  3. Backend creates a Run record, starts asyncio.create_task(run_simulation(...))
  4. Simulator walks the scenario tree:
     a. Creates step with status="running" → broadcasts via WebSocket
     b. Sleeps for configured delay (simulates processing)
     c. Updates step to status="completed" (or "failed") → broadcasts again
     d. Recurses into child steps
  5. Frontend RunWebSocket receives step_update messages
  6. useUpsertStep() merges into TanStack Query cache (no refetch needed)
  7. React re-renders: DAG graph grows, nodes change color, stats update
  8. When all steps are done, run status → "completed", broadcast run_update
  9. Frontend updates header stats, sidebar status dot, enables analysis

9. AI INTEGRATION (Claude / Anthropic)
────────────────────────────────────────────────────────────────────────────────

Tenor uses Claude claude-sonnet-4-20250514 for three AI-powered features:

9.1 Optimization Chat Advisor (chat.py)
  - Persistent chat panel on every page
  - Context-aware: automatically includes current run's step data
  - System prompt instructs Claude to analyze parallelization, cost,
    latency, error handling, token efficiency, and architecture
  - Responses include specific references to step names, token counts, and costs

9.2 Run Performance Analysis (analysis.py → analyze_run)
  - One-click analysis from the "AI Analysis" tab
  - Claude produces a structured JSON report with 7 fields:
    summary, token_analysis, cost_analysis, latency_analysis,
    error_analysis, recommendations[], score (0–100)
  - Score rating: 90-100 Excellent, 70-89 Good, 50-69 Fair, 0-49 Needs Work

9.3 Step Summarization (analysis.py → summarize_step)
  - Translates raw step I/O into plain-English explanations
  - Claude produces: plain_summary, input_summary, output_summary, performance_note
  - Accessible per-step from the inspector panel

All AI features require the ANTHROPIC_API_KEY environment variable to be set.
If missing, the system gracefully degrades with setup instructions instead of errors.

13. KEY DESIGN DECISIONS
────────────────────────────────────────────────────────────────────────────────

  1. Universal Contract: The Run/Step data model is framework-agnostic, allowing
     Tenor to observe any agent system that emits steps in this format.

  2. In-Memory Store: Development simplicity. The Database class interface is
     designed for drop-in replacement with SQLAlchemy + PostgreSQL.

  3. WebSocket per Run: Clients subscribe to individual run rooms, minimizing
     unnecessary broadcasts. The ConnectionManager handles room lifecycle.

  4. Simulation Engine: Instead of requiring a real agent, the simulator walks
     predefined scenario trees with realistic delays, token counts, and errors.
     This enables instant demos without external API costs.

  5. CSS Variable Theming: All colors go through Tailwind CSS custom properties
     rather than hardcoded hex values, enabling seamless light/dark switching.

  6. TanStack Query Cache: WebSocket updates are merged directly into the query
     cache, providing instant UI updates without re-fetching from the API.

  7. dagre Layout: Automatic DAG layout ensures the graph is always readable
     regardless of the number of steps or tree depth.

  8. Claude Integration: Using claude-sonnet-4-20250514 for all AI features, with async
     calls and graceful degradation when the API key is not configured.

  9. Monorepo with pnpm Workspaces: Single repository, shared tooling, independent
     deployment of frontend and backend.

  10. Next.js App Router: Leverages React Server Components, nested layouts,
      and file-based routing for clean code organization.
